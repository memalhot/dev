6. Secure Coding Practices#

6.1. Getting Started#

Code can’t happen in a vacuum. More often than not, in the real world, you’ll be developing software that interacts with other applications, packages, and, importantly, human users. These interactions present inherent vulnerabilities, particularly if the code handles sensitive data. Take, for example, a payment app that collects customer credit card information. If the app leaves that information exposed to unknown third parties—some of whom could be bad actors, such as hackers—the customers could quickly become victims of theft or even fraud. Understandably, then, users anticipate and desire security when interacting with software. While it’s, ahem, foolhardy for a software developer to assume their code is impenetrable, they can follow secure coding practices throughout their development process to reduce the ways bad actors could misuse the program.

6.2. So what even are secure coding practices?#

Secure coding practices are methods, protocols, and decisions that you can use to minimize security vulnerabilities while building software. Think of all the different ways you can develop a program while still maintaining the same purpose. Secure coding practices guide you to develop the program in the most defensive manner possible at any given moment during the process. Let’s take a look at the 14 secure coding practices outlined by OWASP, an open-source nonprofit project dedicated to improving the security of software. These practices are a popular international standard.

6.3. Security By Design#

This first one might seem obvious, or even like—uh oh—circular logic: “Secure your code by securing its design! Make good choices by choosing good choices! Boil water by boiling water!” But as we noted before, prioritizing security throughout the entirety of the design process is integral to minimizing security vulnerabilities. Moreover, security is usually not the only priority for developers, especially in commercial settings. Short-term priorities like speed, efficiency, and scalability are sometimes at odds with and can eclipse security. However, time and time again, we see that “security by design” pays off in the long run with fewer risks of exploitation and damage by bad actors. So as you develop your software, analyze your source code and create proper security automation to detect cyberthreats.

6.4. Input Validation and Output Encoding#

We’re combining these two practices, which basically say the same thing: know thy data. On a trusted system (this means server side), identify everything you can about the program’s input data and sources through a centralized process, and when in doubt, validate it! Classify all input sources as trusted or untrusted, and validate untrusted sources. Likewise, validate the range, length, and character sets (ex: UTF-8) of input data, and add additional security protocols and discrete checks for the suspicious stuff, a.k.a. unidentifiable data. Lastly, canonicalization can be a helpful tool for addressing obfuscation attacks.

Now, for the output side. Once again, through a trusted system (server side), encode your output using standard, tested routines using specified character sets, and sanitize all untrusted data for queries and operating system commands.

6.5. Passwords and Authentication#

Ya know, *********** Passwords and authentication. We’ve all been there. You know, when you’re making an online account for something, and you have to create a password with at least eight characters, including a number, and a capital letter, and a special character, and the secret to the universe (the answer is 42). And then you have to re-enter the password to prove that you have dextrous thumbs and a better memory than a goldfish, but at least one of those two facts is false about you, so you have to erase and retype your password because you don’t know where you messed up because all you can see are asterisks or those weird dots . And this process steals only about two minutes, if that, from your precious life, but you still can’t help but think: remember the good old days of passwords when ‘password’ would suffice? Or better yet, 1234?

Ah, yes. Well. Security practices improve. Life goes on. C’est la vie.

But in all seriousness, password protection and authentication are a large component of secure coding practices. We love passwords and authentication here. In fact… give me a P! Give me an A! Give me an S—no, we’re not doing that? Yeah, okay. Fair enough. I’m surprised you let me get that far. Anywho, here’s a rundown of the tips and tricks you need for securely designed password management and authentication controls:

When in doubt, use a standardized and tested authentication protocol—like a password—for users to access a web page unless that page is meant to be fully accessible to the public, and develop that protocol on a trusted (read: server side) system.

For passwords, set adequate standards of complexity and length (hint: 1234 isn’t good enough). Obscure the password for users as they type it, and, yes, have users confirm their password. Use proper hashing and cryptography—strong, one-way salted hashes—when storing password information server side, and make sure authentication information is securely stored. Moreover, don’t let users reuse the same password particularly if they are doing a password reset, and if they reset their password, alert the user of that reset. Also, set a standard threshold for the number of attempts a user can enter an incorrect password before disabling the account. Don’t let users change their password for at least one day after they create their account. And if you send new users temporary passwords, ensure the password has a short expiration time, and force users to change the password during their first login. If a user must perform a critical operation in their account, have them re-authenticate themselves by password or some other means. Lastly, passwords are not enough for highly sensitive or transactional accounts. Use multi-factor authentication for returning users.

6.6. Session Management#

Any time a user logs into their account, they have begun a “session.” Developers must manage their software’s session controls, or chaos shall reign. A few guidelines:

Firstly, manage and identify these sessions on your own server or framework, and if a user tries to begin a session outside of that framework, treat their session as invalid. Use a randomized algorithm to control sessions and manage users. Establish at least one new session with each login, and maintain HTTPS connection (as opposed to HTTP) throughout the session. Do not allow the same user to log in concurrently. And when a user logs out, the session should completely terminate.

6.7. Access Control#

Two words: default deny. When in doubt, assume that whatever information a user is trying to access is, ahem, above their pay grade, and limit their access as such. Only allow authenticated users to access any non-public information. And if the application cannot access its secure configuration information, do not allow any users to access the application. Finally, ensure the application’s access controls comply with all business rules and flows while still protecting against potential attacks.

6.8. Cryptographic Practices#

Cryptography, you know, the funky symbols you see when a hacker gets into the mainframe in any tv show/movie since 1982 (that year is purely a guess). Essentially, use cryptography to protect any sensitive information from users, and implement all cryptographic practice on a trusted system. Use cryptographically-based randomized number generators to generate random numbers and strings. And manage your cryptographic keys under a secure, standardized process and policies.

6.9. Error Handling and Logging#

Errors. They’re inevitable in life, especially in software development, the most life-y-ist life activity of them all. As you might have heard before, the best way to deal with software errors is to handle and log them. But error handling and logging can introduce new vulnerabilities in addition to any bugs that might already exist in the code. So we must—you guessed it—handle and log errors securely. How do we securely handle and log errors? Well, I’m so glad you asked!

Firstly, when in doubt, log it. Log all errors, exceptions, failures, and potential tampering events. And write and manage these through cryptographic hashing. When responding to errors, do so through a trusted system, and do not reveal or store any sensitive information, including session details, identifiers, account information, and debugging and stack-trace information. Lastly, when there is an error, free all allocated memory from that error.

6.10. Data protection#

Similarly to default deny, use least privilege! Keep users from seeing any data or information unessential to their usage or functioning. Store all sensitive data and information with cryptographic practices, and remove all sensitive data and information when you no longer need it. And remember, access controls are your friend!

6.11. Communication security#

Transmit all sensitive information via encryption, especially Transport Layer Security, or TLS. TLS acts as a protective layer for transported information and should protect information as such. Use TLS for all sensitive content and external system communications. If TLS fails, ensure the backup connection is also secure.

6.12. System Configuration#

System configurations. The nuts and bolts of a system. Figuratively, or should I say, configuratively. But in all seriousness, we’re talking about software here, not hardware. Can you imagine handling an actual bolt? It boggles the mind!

Moving on.

For all software systems, make sure the servers, frameworks, and other components of the system are running on their latest approved versions, and be sure to patch up any vulnerabilities and bugs you can find in these versions. Remove all unnecessary functionality and files, including test code. Define the HTTP methods the system will use (GET or POST) and how the system will use them, and keep the usage of and transference of sensitive information via HTTP to a minimum. Lastly, manage all components and record any changes made to the system configurations.

6.13. Database security#

Chances are, if your software system is storing enough data, you’re going to need a database. Follow these tips and tricks to keep your database secure!

Firstly, consider the database to hold sensitive information. Handle access and the information stored in the database as you would with other sensitive information. That is, create authentication procedures, use secure credentials for database access, and make sure your application operates at the lowest level of privilege when accessing the database. Additionally, strongly type all parameterized queries and variables. That is, be specific with the types of data the queries and variables are holding. Use encryption when handling connection strings. Disable and change any initial default database information provided, and turn off and remove all unnecessary database functionality and default content. Lastly, maintain different connection credentials for different levels of user trust distinctions (ex: read-only user vs. user vs. guest).

6.14. File management#

Sometimes, you’ll need files from users. When that happens, there are some ways to manage those files securely. Firstly, limit the amount of files necessary for upload, as well as the types of files allowed for upload (for example: PDF). Use file headers to validate those uploaded files. Turn off execution privileges in the directories where you store uploaded files, and never reveal the absolute file paths of those uploaded files to users. And of course, scan all uploaded files for viruses and malware.

6.15. Memory management#

Ah, memories! Well, memory in this case. This is our last secure coding practice of the hour. Poetic, isn’t it?

Follow these tips and tricks to handle memory with care! Keep buffering controlled. Don’t use functions that are known to be vulnerable. Be sure to close all resources yourself—garbage collection is snazzy, but don’t rely on it for everything. Lastly, properly free all allocated memory, and overwrite any sensitive information that allocated memory may contain at all exit points in a given function.

6.16. The Finale (Wrapping Up and Other Resources)#

All right, folks. That’s a wrap! We’ve covered 13 points of secure coding practices to help you protect your software: Security by design Input validation and output encoding Passwords and authentication Session management Access control Cryptographic practices Error handling and logging Data protection Communication security System configuration Database security File management Memory management

While this is a start, there is still so much more to the wonderful world of secure coding practices! This article was based off of OWASP’s Secure Coding Checklist, which can give you more detailed information on these thirteen practices. Additionally, check out the OWASP Developer Guide for an even more complete look at security and coding.

Happy coding, and stay snazzy!
